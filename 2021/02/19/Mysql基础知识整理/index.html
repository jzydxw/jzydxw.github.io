***对MySQL数据库进行整体的知识回顾，后期对每个单独技术点及底层原理进行深入学习***

数据库定义

数据库：物理操作文件系统或其他形式文件类型的集合；

实例：MySQL数据库由后台线程以及一个共享内存区组成；

  1. 我们无法直接操作数据库，通过数据库实例来操作数据库文件，类似于操作接口，启动一个MySQL实例会产生两个进程，mysqld是数据库服务守护进程，mysqld_safe是一个用于检查和设置mysqld启动的控制程序，负责MySQL进程的执行，当 mysqld 发生错误时，mysqld_safe 会对其状态进行检查并在合适的条件下重启。
  2. 第一层：用于连接、线程处理；第二层：MySQL核心业务，包括对sql的解析、分析、优化和缓存，存储过程、触发器、视图等；第三层：负责数据的存储和提取的存储引擎

数据的存储

以Innodb存储引擎为例，所有的数据被逻辑地存放在表空间中，表空间（tablespace）–段（segment）–区（extent）–页（page）–行（row）

  1. 同一个数据库实例的所有表空间有相同的页大小，默认页大小为16kb，可以通过innodb_page_size进行修改，也会影响最终区的大小。
  2. innodb存储引擎中，一个区的大小最小为1MB，页的数量最小为64个。
  3. 使用Innodb存储表时，将表的定义和数据索引等信息进行分开存储，表的定义存在.frm文件中，而数据索引存储在.ibd文件中
  4. .frm文件，无论是哪个存储引擎，MySQL表都会在硬盘上创建该文件来描述表的格式或定义
  5. .ibd文件是表空间文件，里面存储了当前表的数据与索引数据。
  6. 页是磁盘管理的最小单位，数据按照行来存储，每个16KB的页中可以存放2-200行的记录。
  7. 这里的行存储方式有两种，一种是倒序存放一行数据中列长度，一种是存放每一列的偏移量（****后续深入了解）

索引

  1. Innodb存储引擎绝大多数使用B+树建立索引，但并不能找到一个给定键对应的具体值，只能找到数据行对应的列，然后把整个页读入到内存中，并在内存中查找具体的数据行。

  2. B+树是平衡树，它查找任意节点耗费时间都是完全相同的，比较次数是B+树的高度。

  3. B+树索引可以分为聚集索引（clustered index)和辅助索引（secondary index）,区别在于，聚集索引中存放着一条行记录的全部信息，而辅助索引值包含索引列和一个用于查找对应行记录的书签。

  4. 聚集索引：按照键的顺序存放，即按照表中主键的顺序构建一颗B+树，并在叶节点中存放表中的行记录数据。所有正常的表应有且只有一个聚集索引（一般为主键），表中的所有行记录数据都是按照聚集索引的顺序存放。

  5. 辅助索引：一张表一定包含一个聚集索引构成的B+树以及若干辅助索引构成的B+树。查询过程：通过辅助索引查找到对应的主键，最后在聚集索引中使用主键获取对应的行记录。

  6. 索引的设计（后续深入了解）

  7. MyISAM和InnoDB两种存储引擎的索引对比

    MyISAM类型的表强调的是性能，其执行数度比InnoDB类型更快，但是不提供事务支持，而InnoDB提供事务支持以及外部键等高级数据库功能。

    • MyISAM引擎使用B+树作为索引结构，叶节点的data域存放的是数据记录的地址。

    • 虽然InnoDB也使用B+Tree作为索引结构，但具体实现方式却与MyISAM截然不同。

      第一个重大区别是InnoDB的数据文件本身就是索引文件。从上文知道，MyISAM索引文件和数据文件是分离的，索引文件仅保存数据记录的地址。而在InnoDB中，表数据文件本身就是按B+Tree组织的一个索引结构，这棵树的叶节点data域保存了完整的数据记录。这个索引的key是数据表的主键，因此InnoDB表数据文件本身就是主索引。

      第二个与MyISAM索引的不同是InnoDB的辅助索引data域存储相应记录主键的值而不是地址。换句话说，InnoDB的所有辅助索引都引用主键作为data域。

  8. 导致索引失效的可能情况：

    • 隐式转换导致索引失效，如varchar当做了number类型。

    • 对索引列进行运算导致索引失效,进行运算包括(+，-，*，/，! 等)

    • 以下使用会使索引失效，应避免使用；

      a. 使用 <> 、not in、not exist、!= 、or、is null 、is not null

      b. like “%_”百分号在前（可采用在建立索引时用reverse(columnName)这种方法处理）

      c. 单独引用复合索引里非第一位置的索引列.应总是使用索引的第一个列，如果索引是建立在多个列上,只有在它的第一个列被 where子句引用时，优化器才会选择使用该索引。

      d.字符型字段为数字时在where条件里不添加引号.

      e.当变量采用的是times变量，而表的字段采用的是date变量时.或相反情况。

      f.不在索引上做任何操作(计算,函数,(自动/手动)类型转换,这样会导致索引失效而转向全表扫描。

      g.避免select *。

    • 不要在 SQL 代码中使用双引号。

锁

  1. Innodb存储引擎采用悲观锁，按照粒度也可以划分成行锁和表锁。

  2. 乐观锁不是锁，只是一种思想，在对资源进行修改的时候，会判断资料是否进行了改变，没有改变就修改，否则就重试，并没有对数据库进行加锁。

  3. 悲观锁是一种锁，在获取资源前对资源加锁，确保同一时刻只有有限的线程能够访问，其他线程都进入等待状态，直达该线程完成操作并释放锁。

  4. 乐观锁与悲观锁的选择：当冲突频率和重试成本较高时推荐使用悲观锁，当需要非常高的响应速度且高并发量的时候推荐使用乐观锁。、

  5. 行级锁：主要分为共享锁（shared lock）和互斥锁（exclusive lock），共享锁（读锁）：允许事务对一条行数据进行读取，互斥锁（写锁）：允许事务对一条行数据进行删除或更新。即共享锁兼容，互斥锁与其他任意锁不兼容。

  6. 表锁：上述的共享锁和互斥锁都是对某一个数据行进行加锁，为了支持多粒度锁定，Innodb存储引擎加入了意向锁（intention lock），这是一种表级锁，意向锁也分为两种：意向共享锁：事务想要在获得表中某些记录的共享锁，需要在表上先加意向共享锁；意向互斥锁：事务想要在获得表中某些记录的互斥锁，需要在表上先加意向互斥锁；意向锁其实不会阻塞全表扫描之外的任何请求，它们的主要目的是为了表示是否有人请求锁定表中的某一行数据。

  7. 锁的算法：record lock、gap lock、next-key lock。

  8. Record Lock:这是一个加到索引记录上的锁，在where语句的过滤条件中，如果是主键过滤，那么innodb可以通过索引建立B+树找到行记录并添加索引，而如果不是主键过滤，innodb不知道待修改的记录具体存放位置，从而锁定整个表。

  9. Gap Lock：间隙锁，即对索引记录中的一段连续区域的锁，当使用类似 SELECT * FROM users WHERE id BETWEEN 10 AND 20 FOR UPDATE; 的 SQL 语句时，就会阻止其他事务向表中插入 id = 15 的记录，因为整个范围都被间隙锁锁定了。间隙锁是存储引擎对于性能和并发做出的权衡，并且只用于某些事务隔离级别。虽然间隙锁中也分为共享锁和互斥锁，不过它们之间并不是互斥的，也就是不同的事务可以同时持有一段相同范围的共享锁和互斥锁，它唯一阻止的就是其他事务向这个范围中添加新的记录。

    事务与隔离级别

    ACID四大特性：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）和持久性（Durability）

原子性：一个事务的所有操作，要么全部成功，要么都不成功。

一致性：事务执行前后保证数据的完整性。比如转账，总金额不变

隔离性：一个事务的执行，不能受其他事务的干扰。

持久性：一个事务一旦执行成功，它对数据的修改就是永久性的。

1. 四种隔离级别：`READ UNCOMMITED`、`READ COMMITED`、`REPEATABLE READ` 和 `SERIALIZABLE`；每个事务的隔离级别其实都比上一级多解决了一个问题。

   - `RAED UNCOMMITED`：使用查询语句不会加锁，可能会读到未提交的行（Dirty Read）；
   
   - `READ COMMITED`：只对记录加记录锁，而不会在记录之间加间隙锁，所以允许新的记录插入到被锁定记录的附近，所以再多次使用查询语句时，可能得到不同的结果（Non-Repeatable Read）；
   
- `REPEATABLE READ`：（MySQL默认事务隔离级别）多次读取同一范围的数据会返回第一次查询的快照，不会返回不同的数据行，但是可能发生幻读（Phantom Read）；
   
   - `SERIALIZABLE`：InnoDB 隐式地将全部的查询语句加上共享锁，解决了幻读的问题；
   
     >脏读：读到另一个事务未提交的数据。
     >
     >不可重复读：对一个事物两次读取不一致，另一个事物做了修改。
     >
     >幻读：对一张表两次查询不一致，另一个事务插入或删除了记录，出现幻行、。

数据库三范式

第一范式：确保每一列的原子性，字段不可再分。

第二范式：满足第一范式，非主键字段依赖于主键字段，一个表只能说明一个事物。

第三范式：满足第二范式，任何非主键属性不能依赖于其他非主键属性。

MySQL基础命令操作

数据库操作

  • 查看数据库：show databases;
  • 创建数据库：create database 名字 charset utf8;并指定编码
  • 使用数据库：use 名字;
  • 改变数据库编码：alter database 名字 charset utf8;
  • 删除数据库：drop database;

表操作

  • 创建表：create table 名字 (id int, name char(16));
  • 查看表：show tables;
  • 查看刚刚创建的表： show create table 名字;
  • 查看表的基本信息：desec 名字;
  • 修改表的数据类型：alter table class modify 名字 char(16);
  • 修改表的字段名和数据类型：alter table class change 名字 新名字 char(18);
  • 新增记录：insert into 名字 values(1,’sky’),(2,’rabbit’);
  • 查询记录：select * from 名字;
  • 修改记录：update 名字 set name = ‘新的值’;
  • 删除记录：delete from 名字 where 条件;
  • 删除表：drop table 名字;

客户端相关操作

  • 启动客户端：mysql -u 账户名 -p
  • 查看字符集编码的命令：show variables like ‘“%char%”;
  • 客户端连接服务端完整命令：mysql -h ip地址 -P 端口 -u 账户 -p 密码;
  • 修改密码：1. 登录后–set password for 用户名@localhost = password(‘新密码’);
                2. mysqladmin -u用户名 -p旧密码 password 新密码

drop 、truncate、delete

drop直接删掉表， truncate删除表中数据，再插入时自增长id又从1开始 ；delete删除表中数据，可以加where字句，delete的自增长id不会变，从上一个最大值开始

当你不再需要该表时， 用 drop；当你仍要保留该表，但要删除所有记录时， 用 truncate；当你要删除部分记录时（always with a WHERE clause), 用 delete.
